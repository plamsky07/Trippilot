# Jira Backlog - Платформа за Събития и Екскурзии

## Основа на проекта

Тема: `Платформа за събития, ученически екскурзии и организирани участия`

Роли:

- `Student` - търси събития и се записва
- `Organiser` - създава и управлява събития
- `Admin` - следи потребители, контрол и статистика

Задължителни изисквания:

1. Системата има login с роли и ролев достъп
2. Има CRUD операции за основни обекти
3. Има търсене и филтриране
4. Има страница за детайли
5. Има audit log кой, кога и какво е правил
6. Има минимум 2 спринта: `MVP` и `Допълнения`
7. Има DoD:
   - изпълнява acceptance criteria
   - има handle на errors
   - има запис и четене от базата
   - има поне 1 работещ тест

## Точни Jira Заглавия

## Epic 1 - Потребителски акаунти

Заглавие за Epic:

- `Epic 1 - Потребителски акаунти`

### User Story 1

Заглавие за Story:

- `US1 - Регистрация на нов потребител`

Описание:

Като нов потребител искам да се регистрирам, за да използвам платформата.

Точни Jira заглавия за задачите:

- `US1 / Task 1 - Създаване на форма за регистрация`
- `US1 / Task 2 - Полета за име, имейл, парола и роля`
- `US1 / Task 3 - Проверка за валиден имейл`
- `US1 / Task 4 - Проверка за минимална дължина на паролата`
- `US1 / Task 5 - Запис в база данни`
- `US1 / Task 6 - Показване на съобщение при успешна регистрация`

Acceptance criteria:

- Потребителят може да въвежда име, имейл, парола и роля.
- При невалиден имейл се показва грешка.
- При парола под минималната дължина се показва грешка.
- При успешна регистрация акаунтът се записва в базата.
- При успешна регистрация се показва съобщение за успех.

### User Story 2

Заглавие за Story:

- `US2 - Вход в системата`

Описание:

Като потребител искам да влизам в системата, за да имам достъп до функционалностите й.

Точни Jira заглавия за задачите:

- `US2 / Task 1 - Създаване на login форма`
- `US2 / Task 2 - Проверка на имейл и парола`
- `US2 / Task 3 - Пренасочване към начална страница`
- `US2 / Task 4 - Съобщение при грешни данни`
- `US2 / Task 5 - Създаване на user session`

Acceptance criteria:

- Регистриран потребител може да влезе с валидни данни.
- При грешен имейл или парола се показва съобщение за грешка.
- При успешен вход се създава активна сесия.
- След вход потребителят се пренасочва към началната страница.
- Ролята на потребителя определя какви действия вижда и може да изпълнява.

## Epic 2 - Управление на събития

Заглавие за Epic:

- `Epic 2 - Управление на събития`

### User Story 3

Заглавие за Story:

- `US3 - Създаване на събитие`

Описание:

Като организатор искам да създам събитие, за да могат учениците да се записват.

Точни Jira заглавия за задачите:

- `US3 / Task 1 - Форма за създаване на събитие`
- `US3 / Task 2 - Полета за име, описание, дата, място, категория и капацитет`
- `US3 / Task 3 - Валидация на задължителните полета`
- `US3 / Task 4 - Запис на събитието в база данни`
- `US3 / Task 5 - Показване на събитието в списъка със събития`

Acceptance criteria:

- Организаторът може да създава събитие.
- Датата не може да бъде в миналото.
- Капацитетът трябва да бъде положително число.
- След успешно създаване събитието се записва в базата.
- Новото събитие се вижда в списъка със събития.

### User Story 4

Заглавие за Story:

- `US4 - Редакция на събитие`

Описание:

Като организатор искам да редактирам събитие, за да коригирам информацията при нужда.

Точни Jira заглавия за задачите:

- `US4 / Task 1 - Бутон редактирай`
- `US4 / Task 2 - Зареждане на текущите данни`
- `US4 / Task 3 - Запис на промените`
- `US4 / Task 4 - Проверка за валидни стойности`
- `US4 / Task 5 - История за последната промяна`

Acceptance criteria:

- Организаторът може да редактира вече създадено събитие.
- Формата се зарежда с текущите данни на събитието.
- При невалидни стойности се показва грешка.
- След запис промените се виждат в детайлната и списъчната страница.
- Последната промяна се пази в audit log или история на промените.

### User Story 5

Заглавие за Story:

- `US5 - Изтриване на събитие`

Описание:

Като организатор искам да изтрия събитие, за да махна отменено такова.

Точни Jira заглавия за задачите:

- `US5 / Task 1 - Бутон изтрий`
- `US5 / Task 2 - Потвърждение преди изтриване`
- `US5 / Task 3 - Изтриване от базата`
- `US5 / Task 4 - Премахване от списъка със събития`
- `US5 / Task 5 - Обновяване на страницата след изтриване`

Acceptance criteria:

- Организаторът може да изтрие свое събитие.
- Системата изисква потвърждение преди изтриване.
- Изтритото събитие се премахва от базата.
- Изтритото събитие вече не се вижда в списъка.
- Действието се записва в audit log.

## Epic 3 - Записване за събития

Заглавие за Epic:

- `Epic 3 - Записване за събития`

### User Story 6

Заглавие за Story:

- `US6 - Записване за събитие`

Описание:

Като ученик искам да мога да се запиша за събитие, за да участвам в него.

Точни Jira заглавия за задачите:

- `US6 / Task 1 - Бутон запиши се`
- `US6 / Task 2 - Проверка за свободни места`
- `US6 / Task 3 - Проверка дали потребителят вече е записан`
- `US6 / Task 4 - Създаване на регистрация`
- `US6 / Task 5 - Обновяване на броя участници`

Acceptance criteria:

- Потребителят може да се запише само веднъж за едно събитие.
- Не може да се запише, ако няма свободни места.
- След записване се показва съобщение за успешно регистриране.
- Регистрацията се записва в базата.
- Броят участници се обновява автоматично.

### User Story 7

Заглавие за Story:

- `US7 - Отписване от събитие`

Описание:

Като ученик искам да мога да се отпиша от събитие, за да освободя мястото си.

Точни Jira заглавия за задачите:

- `US7 / Task 1 - Бутон отказвам участие`
- `US7 / Task 2 - Премахване на регистрация`
- `US7 / Task 3 - Обновяване на свободните места`
- `US7 / Task 4 - Потвърждение за отказ`

Acceptance criteria:

- Ученикът може да се отпише от събитие, за което е записан.
- Системата иска потвърждение преди отказ.
- След отказ свободните места се обновяват.
- Регистрацията се премахва или маркира като отказана в базата.
- На екрана се показва съобщение за успешно отказване.

### User Story 8

Заглавие за Story:

- `US8 - Преглед на списък с участници`

Описание:

Като организатор искам да виждам списъка с участници, за да виждам кой ще присъства.

Точни Jira заглавия за задачите:

- `US8 / Task 1 - Страница с участници`
- `US8 / Task 2 - Таблица с име и имейл`
- `US8 / Task 3 - Брой записани участници`
- `US8 / Task 4 - Търсене в списъка`
- `US8 / Task 5 - Сортиране по име`

Acceptance criteria:

- Организаторът вижда списък с всички записани участници.
- Таблицата показва поне име и имейл.
- Броят записани участници е видим и коректен.
- Търсенето в списъка работи по име или имейл.
- Сортирането по име работи коректно.

### User Story 11

Заглавие за Story:

- `US11 - Преглед на всички потребители от администратор`

Описание:

Като администратор искам да мога да виждам всички потребители, за да управлявам платформата.

Точни Jira заглавия за задачите:

- `US11 / Task 1 - Таблица с потребители`

## Definition of Done

Една задача или user story е `Done`, когато:

1. Изпълнява acceptance criteria
2. Има handle на errors
3. Има запис и четене от базата
4. Има поне един работещ тест
